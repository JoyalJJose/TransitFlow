# Technology Selection and Justification

This document justifies each major technology selected for TransitFlow from a research and final-year-project perspective. Each entry links the choice to concrete project requirements and to the broader body of work around real-time public-transit systems, Internet-of-Things messaging, and edge machine learning. The discussion is deliberately written to be citation-ready: specific references can be slotted in during the report-writing phase without restructuring the argument.

## Introduction

Technology selection for a final-year project is itself a research activity: each choice must be justifiable against the project's objectives, situated within the relevant literature, and weighed against credible alternatives rather than adopted by default. The selection process applied here began by deriving functional and non-functional requirements from the project brief and narrowing the candidate space to tools and standards that appear repeatedly in peer-reviewed work on real-time systems, Internet-of-Things messaging, and applied machine learning. Shortlisted candidates were then assessed against consistent criteria - fitness for purpose, maturity and community support, licensing, interoperability with the rest of the stack, and reproducibility of results - before a final selection was made.

Three cross-cutting principles guided every decision. First, preference was given to open standards and open-source implementations so that the work remains reproducible and independently verifiable. Second, preference was given to technologies that are well represented in the academic literature, so that implementation choices can be argued from prior work rather than from first principles. Third, transparent and inspectable components were preferred over heavy abstractions, so that the resulting artefact exposes its data flows and control paths clearly enough to be evaluated. The sections that follow apply these criteria to each technology in turn, in each case describing what the technology is, how it is used in this project, and why its selection is defensible for a research-oriented final-year project.

## Per-Technology Justification

### MQTT (paho-mqtt v5, Eclipse Mosquitto, TLS on 8883)

MQTT is widely regarded as the de-facto lightweight publish/subscribe protocol for constrained IoT deployments, with a compact fixed header, a broker-mediated topology, and well-understood Quality-of-Service semantics. TransitFlow uses MQTT as the single control and telemetry fabric between edge devices and the backend, driven by `paho-mqtt>=2.0` in [src/Edge/comms.py](../src/Edge/comms.py) and [src/Backend/MQTTBroker/broker_handler.py](../src/Backend/MQTTBroker/broker_handler.py), with an Eclipse Mosquitto broker configured in [docker/mosquitto/mosquitto.conf](../docker/mosquitto/mosquitto.conf). The project deliberately adopts MQTT v5 so that reason codes, session-expiry properties, and the `no_local` subscription option can be used; the unified `edge/{id}/model` topic depends on `no_local` to prevent self-echo of chunked model artefacts. Persistent sessions (`clean_start=False` with a maximum `SessionExpiryInterval`) combined with paho's internal offline queue give first-class tolerance of intermittent connectivity, which is an explicit requirement of the edge scenario. Security is enforced by TLS on port 8883 with a CA-signed server certificate and mutual client certificates generated via [docker/mosquitto/certs/generate_certs.sh](../docker/mosquitto/certs/generate_certs.sh), giving an auditable, realistic trust model appropriate for a research artefact that must be defended.

### PyTorch (torch 2.9.1 / torchvision 0.24.1)

PyTorch is now the dominant framework in contemporary computer-vision and crowd-analysis literature, used both for published reference implementations and for widely distributed pre-trained weights; its dynamic computation graph is also broadly cited as an advantage during experimental development. TransitFlow uses PyTorch indirectly through the Ultralytics YOLO runtime, but the framework is an explicit top-level dependency declared in [src/Edge/requirements.txt](../src/Edge/requirements.txt) and exercised by `CrowdCounter` in [src/Edge/inference.py](../src/Edge/inference.py), which selects a CUDA device where available and falls back to CPU otherwise. The dual-target posture is important for a final-year project: the same inference code runs reproducibly on a commodity development laptop and on a GPU-equipped edge board, allowing the evaluation chapter to compare throughput and latency across platforms without forking the codebase. Using PyTorch rather than a bespoke or framework-agnostic runtime also preserves a clear upgrade path to published detection and regression models should the scope of the thesis later extend beyond counting.

### Ultralytics YOLO (ultralytics 8.3.228)

The YOLO ("You Only Look Once") family is the canonical real-time object-detection architecture in the literature and, for person detection specifically, typically achieves a favourable accuracy-versus-latency trade-off compared with two-stage detectors. The Ultralytics distribution packages training, validation, and inference behind a uniform `model.predict` API that is reproducible across releases, and it is pinned to a specific version in [src/Edge/requirements.txt](../src/Edge/requirements.txt) to preserve reproducibility of the thesis evaluation. TransitFlow uses Ultralytics YOLO inside `CrowdCounter` in [src/Edge/inference.py](../src/Edge/inference.py) to convert a captured frame into a crowd count using runtime-configurable confidence thresholds and zone metadata. Crucially, the single-file `.pt` artefact produced by Ultralytics is what makes the project's over-the-air model-swap protocol tractable: chunked SHA-256-verified distribution, backup-and-rollback on load failure, and live pause/resume of inference (all coordinated in [src/Edge/model_manager.py](../src/Edge/model_manager.py) and [src/Edge/model_receiver.py](../src/Edge/model_receiver.py)) would be substantially harder against a framework with a multi-file export format.

### OpenCV (opencv-python 4.12)

OpenCV is among the most widely cited image-processing and computer-vision libraries in both academic and industrial practice, which means that methodology built on it is immediately reproducible by other researchers and readily legible to examiners. TransitFlow uses `opencv-python` in [src/Edge/camera.py](../src/Edge/camera.py), where `VideoCapture` provides a uniform abstraction over USB, CSI, and thermal camera devices; this is an important property for an FYP because the physical capture device may reasonably change between development, laboratory testing, and final demonstration without requiring code changes. OpenCV is also used for frame persistence to disk so that inference operates on a stable file path rather than an in-memory buffer, which simplifies logging and post-hoc inspection during evaluation. Pairing OpenCV capture with a size-one `FrameBuffer` guarded by a `threading.Condition` yields a textbook producer-consumer pipeline in which stale frames are overwritten and inference always operates on the most recent image - a design that can be discussed in the report as a deliberate application of established concurrency patterns.

### PostgreSQL + TimescaleDB (timescale/timescaledb:latest-pg16)

The data that TransitFlow must persist is fundamentally two-faceted: it is relational (routes, stops, vehicles, and their joins are central to every dashboard query) and it is strongly time-series (per-second crowd counts, vehicle telemetry, GTFS-RT trip updates, and prediction outputs accumulate monotonically). TimescaleDB is purpose-built for exactly this combination, layering automatic time-based partitioning, continuous aggregates, and native compression over an unmodified PostgreSQL surface. The schema in [docker/database/init.sql](../docker/database/init.sql) exploits this directly by converting `crowd_count`, `stop_logs`, `vehicle_telemetry`, `gtfs_rt_vehicle_positions`, `gtfs_rt_trip_updates`, and `predictions` into hypertables, while keeping the reference tables (`routes`, `stops`, `route_stops`) as ordinary relational tables. For a single-student final-year project, this unification is a deliberately defensive architectural choice: it avoids the polyglot-persistence overhead of operating, for example, a separate time-series database alongside PostgreSQL, and it keeps the entire analytics surface addressable in one SQL dialect, which is essential when the dissertation must document and defend every data path end to end.

### SQL / psycopg2 (psycopg2-binary, asyncpg)

TransitFlow deliberately exposes the database through hand-written, parameterised SQL rather than through an object-relational mapper. The synchronous path uses `psycopg2-binary`, which is the most mature and broadly taught PostgreSQL driver in the Python ecosystem and is used by the ingestion and seeding layers in [src/Backend/Database/writer.py](../src/Backend/Database/writer.py) and [src/Backend/MQTTBroker/broker_handler.py](../src/Backend/MQTTBroker/broker_handler.py); the asynchronous path uses `asyncpg` inside the FastAPI event loop so that the WebSocket fan-out is not blocked by query latency. The concrete query surface lives in [src/Backend/API/queries.py](../src/Backend/API/queries.py), where each dashboard figure is backed by an explicit, inspectable SQL statement. This choice is aligned with the research orientation of the project: every datum displayed on the dashboard can be traced to a concrete query in a concrete file, which is a strong position from which to write the data-flow chapter of the report and is materially easier to verify during assessment than equivalent ORM-generated SQL.

### GTFS-Realtime (gtfs-realtime-bindings / protobuf)

GTFS-Realtime is the de-facto open standard for live public-transit feeds. It is maintained as an open specification by Google together with the wider transit community, is encoded in Protocol Buffers for compactness and schema validation, and is the format in which Ireland's National Transport Authority publishes live Dublin Bus data. TransitFlow consumes this feed in [src/Backend/GTFS_RT/fetcher.py](../src/Backend/GTFS_RT/fetcher.py) using the official `gtfs-realtime-bindings` Python package and persists parsed `TripUpdate` entities (including the extracted `direction_id` from each `TripDescriptor`) into the `gtfs_rt_trip_updates` hypertable. Adopting the standard rather than scraping a proprietary endpoint has two research-relevant consequences: the evaluation is grounded in real, externally verifiable data from a recognised authority, which strengthens the external validity of any claims about the prediction engine's behaviour; and the ingestion pipeline is portable to any other GTFS-RT-compliant agency with only a configuration change, giving the artefact a defensible generalisation story beyond the specific Dublin Bus evaluation.

### FastAPI (fastapi, uvicorn[standard])

FastAPI is an asynchronous Python web framework that is now routinely used in production systems and is increasingly prevalent in research prototypes that require both REST endpoints and streaming protocols within a single process. TransitFlow exposes its entire backend surface through FastAPI in [src/Backend/API/main.py](../src/Backend/API/main.py) and [src/Backend/API/queries.py](../src/Backend/API/queries.py), served by the `uvicorn[standard]` ASGI server. The async-first design pairs naturally with the project's two concurrent workloads: serving the dashboard's WebSocket fan-out without blocking, and consuming PostgreSQL `LISTEN`/`NOTIFY` events from a background thread so that fresh data can be pushed the moment it is committed. In addition, FastAPI generates OpenAPI documentation automatically and integrates Pydantic-typed request and response models, which effectively produces a self-describing API surface; this reduces the amount of hand-written documentation that must appear in the dissertation and gives examiners a canonical endpoint reference without leaving the code base.

### WebSockets

For an operational dashboard whose value is precisely its immediacy, periodic HTTP polling is both wasteful and fundamentally stale: it trades predictable latency for server load and cannot represent event-driven quantities such as alert emission cleanly. TransitFlow therefore uses WebSockets, specifically the `/ws/dashboard` endpoint implemented in [src/Backend/API/main.py](../src/Backend/API/main.py) and backed by the connection manager in [src/Backend/API/ws.py](../src/Backend/API/ws.py). The backend assembles a unified payload via `build_dashboard_payload()` in [src/Backend/API/queries.py](../src/Backend/API/queries.py) whenever the database emits a change notification, and the connection manager performs per-client coalescing so that a slow dashboard instance cannot back up the queue or force the producer to block. This structure allows the report to make a concrete, evaluable engineering claim about backpressure-aware live streaming, rather than merely asserting "real-time" behaviour; it also makes the communication pattern symmetric with the MQTT control plane on the edge side, which simplifies the architectural narrative.

### React + Vite

React is the most widely adopted component-based frontend framework in both industry and academic coursework, and its unidirectional data-flow model maps cleanly onto the problem of binding an incoming event stream (in this case, the backend WebSocket payload) to a reactive user interface. The dashboard in [src/Frontend/dashboard/](../src/Frontend/dashboard/) uses React together with Vite as its development server and build tool; Vite's sub-second hot module replacement materially shortened the iteration loop during implementation, which matters in the constrained time budget of a final-year project. The combination is also pragmatic from an assessment perspective: because React is so ubiquitous, examiners and external reviewers can read and reason about the dashboard source without specialised framework knowledge. Finally, React's ecosystem (including the Mapbox GL bindings used for the map view) meant that domain-specific visual components, such as live stop-level crowd overlays, could be composed from well-documented primitives rather than being reimplemented from scratch.

### Docker + Docker Compose

TransitFlow is intrinsically a multi-service system: an MQTT broker, a TimescaleDB instance, a Python supervisor that runs the MQTT handler, GTFS-RT poller, FastAPI, and prediction engine, a simulator, and a React dashboard must all be brought up in a specific order, over a shared network, with coordinated credentials and healthchecks. Reproducing such a topology manually is prohibitively fragile, especially for an examiner attempting to evaluate the artefact on a fresh machine. The project therefore encodes the entire deployment in [docker/docker-compose.demo.yml](../docker/docker-compose.demo.yml), including explicit `depends_on` ordering with health-gated conditions, hardened security settings (`read_only` root filesystems, `cap_drop: ALL`, `no-new-privileges`), named volumes for persistent state, and a stable set of three image tags under the `transitflow-demo` project namespace. This makes the deployment architecture itself an evaluable contribution: it directly satisfies the FYP requirement of a reproducible artefact, and it permits the report to claim that any reviewer with Docker Desktop can reconstruct the full stack with a single `docker compose up` command.
